## 10. 2026-04-08 SD 字体与竖屏长页面联调补充

### 10.1 今天已验证通过的内容

- SD 卡方案已经跑通，当前更稳的选择是 `32GB + MBR + 单分区 + FAT32`
- ESP32-P4 可以成功挂载 `/sdcard`
- 固件可以在启动时把字体写入 SD 卡
- 运行时 FreeType 可以从 `/sdcard/FONTS` 加载字体
- UI 已切回竖屏默认方向
- 页面已经改成长页面，整体可以上下滑动

今天实机已经确认过的关键日志特征：

- `sdcard mounted at /sdcard`
- `sd asset already present: /sdcard/FONTS/CN_UI.TTF`
- `ui_theme: loaded runtime font: /sdcard/FONTS/CN_UI.TTF`
- `ui_theme: loaded runtime font: /sdcard/FONTS/JP_UI_R.TTC`
- `ui_theme: loaded runtime font: /sdcard/FONTS/SYMBOLS.TTF`

### 10.2 SD 卡与文件系统方面的结论

最初 64GB 卡的主要问题不是容量，而是常见格式为 `exFAT`。
当前工程挂载 SD 使用的是 FAT 路线，因此实操上 `32GB FAT32` 更省事、更稳。

当前建议继续保持：

- 分区表：`MBR`
- 文件系统：`FAT32`
- 字体目录：`/sdcard/FONTS`

### 10.3 中文字体写入失败的根因

一开始中文字体写入失败，不是因为 SD 卡没挂载，而是因为文件名超出了当前 FatFs 允许范围。

实际问题是：

- 当前工程没有开启长文件名支持
- 使用了 `UI_FALLBACK.TTF` 这种超出 8.3 风格的文件名
- 结果 `fopen()` 直接失败

后续已经改成短文件名：

- 中文：`CN_UI.TTF`
- 符号：`SYMBOLS.TTF`

这里得到的经验很明确：

- 只要当前 FatFs 配置不变，文件名就应尽量保持短 ASCII 名
- 不要直接用中文名、日文名、超长名

### 10.4 为什么日语能显示但中文最初不行

这个阶段很容易误判。

当时日语能出来，不代表 SD 字体已经成功生效。更可能是：

- 内置字体本身覆盖了一部分 CJK / 日文字符
- 但中文主字体实际上还没真正写进去或加载成功

所以后面判断字体链路是否生效时，必须同时看：

- 屏幕现象
- 顶部状态提示
- 串口日志

不能只看“某些字出来了”。

### 10.5 中文字体触发 FreeType 栈保护错误的根因

中文字体真正开始加载之后，系统出现过重启循环。

关键 panic 特征：

- `Stack protection fault`
- 落点在 FreeType 的 `cf2_interpT2CharString`

说明问题不在 SD 挂载，不在字体路径本身，而在字体格式和任务栈。

当时的根因组合是：

- 中文字体先用了 `Noto Sans SC (TrueType).otf`
- 这条 OTF/CFF 路线在当前 FreeType 场景下更吃栈
- `lvgl` 任务栈太小

后续处理改成：

- 放弃当前那份 OTF 中文字体
- 改用 `simhei.ttf` 作为中文主字体并内嵌到固件
- 运行时落盘为 `/sdcard/FONTS/CN_UI.TTF`
- 同时把 `lvgl` 任务栈增大，并尝试放入 `PSRAM`

当前经验是：

- 这套平台上，中文字体优先尝试 `TTF/TTC`
- 不要默认优先上 `OTF/CFF`

### 10.6 当前实际采用的字体策略

目前这条链路更接近验证版，但已经是可工作的方向：

- 中文主字体：`CN_UI.TTF`
- 日文字体：`JP_UI_R.TTC`
- 符号字体：`SYMBOLS.TTF`
- Emoji：`EMOJI.TTF`

当前逻辑是：

- 固件内嵌中文和符号字体
- 启动后若 SD 挂载成功，则把它们写到 `/sdcard/FONTS`
- 若文件已存在且大小一致，则跳过重复写入
- 运行时再从 SD 路径加载

这种做法的优点是：

- 开发阶段不需要每次手动拔卡拷文件
- 可以先验证“板子自己写卡 + 自己读卡”的闭环

## 11. 2026-04-09 字体链路瘦身、分页阅读页与性能判断

> 本节覆盖第 10 节里与“固件内嵌字体、长页面滚动测试”有关的后续变化。涉及当前实现时，以本节为准。

### 11.1 今天实际完成的修改

1. 去掉了“把字体编进固件再写入 SD 卡”的中间链路。
2. 当前固件只负责挂载 `/sdcard` 并检查字体文件是否存在，不再负责把字体写回 SD 卡。
3. UI 运行时继续直接从 `/sdcard/FONTS` 读取字体。
4. 阅读页不再做整页连续滚动测试，已经改成分页阅读模式。
5. 页面正文里所有“测试说明、分辨率提示、SD 说明、滚动说明”都已经删除，只保留日文正文和中文翻译。
6. FreeType 字形缓存从 `256` 提高到 `1024`，用于降低重复取字形时的卡顿。

当前字体路径仍是：

- `/sdcard/FONTS/CN_UI.TTF`
- `/sdcard/FONTS/JP_UI_R.TTC`
- `/sdcard/FONTS/SYMBOLS.TTF`
- `/sdcard/FONTS/EMOJI.TTF`

### 11.2 今天遇到的核心问题

#### 11.2.1 为什么固件曾经非常大

根因不是正文文本，而是字体文件被 `EMBED_FILES` 编进了 app 镜像。

当时的链路实际上是：

1. 字体先内嵌到固件。
2. 开机后再把内嵌字体写入 SD。
3. UI 再从 SD 读取这些字体。

这会导致：

- 固件体积非常大。
- SD 已经有字体时，这条链路仍然显得重复。
- 用户容易误以为“字体既然已经在 SD 里，为什么 bin 还是这么大”。

后续已经处理为：

- 删除 `EMBED_FILES`
- 删除启动写卡逻辑
- 仅保留“挂载 SD + 检查文件 + 运行时加载字体”

处理后，固件体积从接近 `14 MB` 量级降到了约 `1.6 MB` 量级，app 分区重新恢复为大量空余。

#### 11.2.2 为什么长文滚动会明显卡顿

滚动卡顿的主因不是“SD 卡慢”这一个因素，而是下面几项叠加：

- 使用了 `SD + FreeType` 运行时字体
- 页面里有大段中文和日文正文
- 触摸拖动时需要持续重排文本、查字形、栅格化、重绘
- 若字形缓存命中率不高，就会不断回到字体文件取新字形

实机现象是：

- 早期 perf 可能从几十毫秒升到上千毫秒
- 调整后已降到大约 `300 ms` 左右
- 体感上仍有明显卡顿，不适合作为“像手机一样刷流”的方案

#### 11.2.3 为什么“下一页”慢、“上一页”快

当前分页版里，这很大概率就是缓存命中差异造成的，属于符合预期的现象。

更具体地说：

- 点击“下一页”时，往往要首次显示一批新的汉字/日文字形。
- 这些字形如果还没被 FreeType/LVGL 缓存过，就要现场从字体文件读取并栅格化。
- 点击“上一页”时，前一页刚刚显示过，相关字形仍很可能还留在 RAM 缓存里，所以会更快。

结论：

- `上一页` 快，通常是因为前一页字形和布局结果刚刚用过。
- `下一页` 慢，通常是因为遇到了新的字形集合。

这个缓存是运行时 RAM 缓存，不是写回 SD 的持久缓存。

因此下一次重新上电时：

- SD 卡里的字体文件当然还在。
- 但 FreeType / LVGL 的字形缓存会重新开始建立。
- 也就是说，首次翻到新页时仍然可能再次出现“第一次慢、回看快”的现象。

### 11.3 今天采用的解决方式

1. 删掉固件内嵌字体，只保留 SD 运行时字体。
2. 删掉启动写 SD 卡逻辑，避免继续增大镜像体积。
3. 清理阅读页里的调试说明文本，减少干扰。
4. 把整页滚动阅读改成分页阅读，避免在拖动过程中持续重排和重绘整页长文。
5. 提高 FreeType 字形缓存数量，降低重复页面切换时的栅格化压力。

### 11.4 当前更合理的判断

如果目标是：

- 在 MCU 上显示中日文长文
- 页面内容主要用于查看说明、记录、手册、日志摘要
- 不追求手机社交软件式连续滚动

那么“分页阅读”比“长页面连续滚动”更现实，也更稳定。

如果目标是：

- 真正做到像手机那样顺滑地刷长文流
- 同时又使用大段中日文和运行时 TTF/TTC 字体

那对当前这块板子和这套 GUI 路线来说，成本会明显偏高，不应作为近期主目标。

### 11.5 下一步建议

建议按优先级这样推进：

1. 先稳定分页阅读页，确认分页切换、字体显示、触摸命中都可接受。
2. 观察“首次下一页较慢”是否只发生在首次访问新页时。
3. 如果要进一步优化翻页首帧，可考虑做“字形预热”或“预翻页预渲染”。
4. 如果后面要做真正的阅读器，不建议继续押注“长文连续滚动 + SD 上动态字体”。
5. 若目标是实用型阅读器，应优先考虑位图字库或离线预渲染方案。

### 11.7 本机如何用 PowerShell 稳定写文件

如果需要用 PowerShell 直接改文件，当前更稳的方式不是 `Set-Content`，而是：

1. 先 `Get-Content -Raw` 读全文。
2. 用 `.Replace(...)` 做精确替换，或用 here-string 准备整段内容。
3. 用 `[System.IO.File]::WriteAllText(...)` 写回。
4. 显式指定 `UTF8Encoding($false)`，避免 BOM 带来的额外问题。

已验证可用的模板：

```powershell
$path = 'components\terminal_gui_app\src\ui\ui_shell.c'
$content = Get-Content -Path $path -Raw
$content = $content.Replace('旧内容', '新内容')
$utf8NoBom = New-Object System.Text.UTF8Encoding($false)
[System.IO.File]::WriteAllText((Resolve-Path $path), $content, $utf8NoBom)
```

如果是追加整段文档，可直接这样写：

```powershell
$path = 'D:\Vscode\ROS2\ROS2-SLAM-ESP32P4-Project\docs\esp32p4.md'
$append = @"
这里放要追加的 Markdown 内容。
"@
$old = Get-Content -Path $path -Raw
$utf8NoBom = New-Object System.Text.UTF8Encoding($false)
[System.IO.File]::WriteAllText($path, $old + $append, $utf8NoBom)
```

这么做的原因是：

- 可控性更高，适合大段文本。
- 不容易因为换行或编码差异把文件写乱。
- 这次会话里已经多次用这种方式稳定修改源码和文档。

### 11.8 位图字库方案的想法

当前已经可以明确区分两条路线：

#### A. 动态栅格化字体

即当前方案：

- SD 卡里放 `TTF/TTC`
- 运行时由 FreeType 按需查找并栅格化字形

优点：

- 灵活
- 一个字体文件可覆盖多个字号
- 文本仍然是文本，适合动态内容

缺点：

- MCU 上 CPU 压力大
- 长文 + CJK + 高频翻页/滚动时性能代价明显
- 首次遇到新字时很容易卡顿

#### B. 位图字库

思路是：

- 提前把固定字号、固定字重的常用字生成位图
- 运行时直接读取位图并显示
- 可以把位图字库放在 SD 卡里，不一定要编进固件

优点：

- 比动态栅格化轻得多
- 更适合 MCU
- 翻页或滚动时性能更可控

缺点：

- 字号不灵活
- 字集一大时，字库体积会明显增长
- 需要自己设计索引、缓存和加载方式

当前更值得考虑的落地方向是：

1. 固定字号位图字库 + SD + 分页阅读
2. 整页预渲染图片 + SD + 分页阅读

如果目标是“实用型 MCU 阅读器”，这两条路线都比“SD 上 TTF/TTC 动态栅格化长文流”更靠谱。

### 11.9 当前结论

截至 2026-04-09，当前项目在阅读场景下更适合的方向已经比较明确：

- 先以分页阅读为主，不再强求手机式长文滚动。
- 字体继续直接从 SD 读取，不再重复写卡，也不再把大字体编进固件。
- 如果后续仍觉得翻页首帧偏慢，应优先考虑预热缓存、位图字库或预渲染，而不是继续死磕连续滚动。

### 11.10 2026-04-14 WiFi(C6 Hosted) 联调补充

这次已经把 ESP32-P4 + 外置 ESP32-C6 的 WiFi 链路跑通，结论如下：

#### 1. 根因一开始不是“扫不到网”，而是 Host 侧传输方式配错了

一开始固件虽然能编过，但运行日志里实际走的是：

- spi_wrapper: Transport: SPI
- uild config: ... sdio=0 cp_c6=0

这对 ESP32-P4-WIFI6-DEV-KIT 是错误方向。

这块板子的正确方向应是：

- ESP32-P4 作为 Host
- ESP32-C6 作为 WiFi/BT 协处理器
- 两者通过 SDIO 通信
- 工程侧要使用 esp_wifi_remote + esp_hosted

这次最终修正后的关键配置是：

- CONFIG_ESP_HOSTED_CP_TARGET_ESP32C6=y
- CONFIG_ESP_HOSTED_SDIO_HOST_INTERFACE=y
- CONFIG_ESP_WIFI_REMOTE_LIBRARY_HOSTED=y

运行时正确日志应类似：

- uild config: remote=1 hosted=1 sdio=1 cp_c6=1
- Identified slave [esp32c6]
- esp_wifi_init ok
- esp_wifi_start ok
- esp_wifi_scan_start ok

#### 2. 如果切了配置还是老样子，优先怀疑 build/sdkconfig 缓存

这次的经验是：

- 光改 sdkconfig.defaults 不够
- 如果 uild 和 sdkconfig 里还残留旧配置，最终刷进去的很可能仍然是旧的 SPI 版

更稳的处理方式：

1. 删除 uild
2. 删除 sdkconfig
3. 保留并修正 sdkconfig.defaults
4. 重新 configure/build

否则很容易出现“看上去改了，实际上刷进去没变”的情况。

#### 3. C6 固件版本提示

当前联调时日志里出现过：

- Version mismatch: Host [2.12.0] > Co-proc [0.0.0]

这说明板上 C6 的 hosted slave 固件版本信息并不理想，但当前链路已经能正常：

- 建链
- 启动 STA
- 扫描 AP

所以这不是当前阶段的主阻塞。

但如果后续出现：

- RPC timeout
- 扫描/连接偶发异常
- 长时间运行不稳定

仍然应把 C6 hosted slave 固件版本核对/升级列为排查项。

### 11.11 2026-04-14 WiFi 扫描崩溃与 GUI 问题补充

#### 1. 扫描完成后蓝屏重启的根因

问题现象：

- esp_wifi_scan_start ok
- 收到 SCAN_DONE
- 随后 sys_evt 任务触发 Stack protection fault

这次确认的根因不是 WiFi 本身，而是 SCAN_DONE 事件回调写得太重：

- 在系统事件线程栈上放了较大的 wifi_ap_record_t records[...] 局部数组
- 同时又在回调里打了较重的日志

修正方式：

- 不要在 SCAN_DONE 回调的局部栈上放大数组
- 把扫描结果缓存改成全局/静态上下文缓冲区
- 减少逐条 AP 的重日志

这次修完后，扫描已可稳定返回 15 个 AP，系统不再因 SCAN_DONE 直接重启。

#### 2. WiFi 页面“点不动/滑不动”的一部分根因其实是布局裁剪，不只是触摸

这次页面问题分成两类：

##### A. 控件根本没显示出来

表面看起来像“按钮点不到”，但实际问题是：

- WiFi 卡片父容器高度没有按内容展开
- 列表、密码框、扫描/连接按钮被卡片裁掉了
- 屏幕上只剩摘要文字，所以自然无从点击

这类问题的修法是：

- 对卡片、状态块、按钮行等容器显式设置 LV_SIZE_CONTENT
- 不要假设 LVGL 容器一定会自动按子控件高度展开

##### B. 下拉框在触屏小屏上体验差

原来用的是 dropdown 选 WiFi。

这在桌面 UI 里问题不大，但在这个项目里更适合：

- 直接做“可滚动 WiFi 列表”
- 每个 AP 一行按钮
- 点击选中后高亮
- 再配合密码输入框和连接按钮

也就是说，在这个项目里：

- dropdown 不是最佳触摸交互
- scrollable list + selected item 更可靠

#### 3. 页面状态不要长期依赖初始化时的静态诊断值

这次还遇到一个容易误导的问题：

- 页面里 WiFi: init failed 这种文案，实际上已经不是真实运行状态
- 因为它来自启动时一次性的 diag 快照，而不是实时 WiFi 状态

经验：

- 启动诊断值适合做“初始提示”
- 运行期显示应优先使用 service snapshot（实时状态）

否则 UI 很容易“实际已经连上了，但文案还显示失败”。

### 11.12 字库文件到底是“下载”还是“自己做”

这个问题要分清两个层次：

#### 1. “字体源文件”不建议从零开始自己画

如果这里说的是：

- 从零设计一整套中文字体
- 自己画字形

那不现实，也没必要。

更合理的做法是：

- 先选一个合法可用的现成字体源文件（TTF/TTC/OTF）
- 再从这个字体里生成项目真正需要的字库包

也就是说：

- 字体“设计”本身通常不自己做
- 但“面向本项目的字库包生成”完全可以自己做

#### 2. 本项目真正应该自己做的是“位图字库打包流程”

更推荐的路线是：

1. 选一个合法字体源文件
2. 指定字号，例如 18px / 20px
3. 指定字符集（UI 文案、菜单、日志常用字、专有词）
4. 在 PC 端把这些字渲染成位图
5. 生成自己的字库包，例如：
   - manifest.json
   - index.bin
   - glyph.bin
6. 通过 WiFi 上传到板子 SD 卡
7. 板子运行时直接查索引并显示位图

这条路线里，真正“自己做”的部分是：

- 字符集裁剪
- 位图字库导出工具
- 字库包格式
- 板子端加载器

这部分是完全值得做、而且也更适合 MCU 项目的。

#### 3. 当前建议

当前最务实的做法是：

- 先用现有 CN_UI.TTF / 其他合法字体作为源
- 不要自己重新设计字体
- 先做“固定字号 + 指定字符集”的位图字库导出工具
- 优先服务 UI 场景，而不是一开始就追求全量中文

也就是说，顺序应是：

- 先把上传链路做通
- 再把字库格式定住
- 最后再考虑字体源文件是否替换

### 11.13 SD 卡高速模式（40 MHz）能不能用，会不会影响当前项目

这次核对当前 BSP 实现后，结论非常明确：

#### 1. 当前项目其实已经在用 SDMMC 高速模式

waveshare__esp32_p4_platform/esp32_p4_platform.c 里的 sp_sdcard_mount() 当前就是：

- sdmmc_host_t host = SDMMC_HOST_DEFAULT();
- host.slot = SDMMC_HOST_SLOT_0;
- host.max_freq_khz = SDMMC_FREQ_HIGHSPEED;
- slot_config.width = 4;

也就是说：

- 当前项目的 SD 卡已经是 4-bit
- 并且已经把频率拉到了 High-speed (40 MHz)

所以对当前工程来说，这不是“要不要开启”的问题，而是“当前 BSP 本来就这么配了”。

#### 2. 手册里给的那组引脚，本质上对应的是 Slot0 的 SD 卡连接

手册里示例给出的：

- clk = 43
- cmd = 44
- d0 = 39
- d1 = 40
- d2 = 41
- d3 = 42

而当前 BSP 注释写的是：

- Slot 0 uses IO MUX, so not specifying the pins here

这两者并不冲突。

含义是：

- BSP 直接使用 Slot0 + IO MUX 默认接法
- 所以没有手动把这些 pin 再逐个写出来
- 但底层实际就是那套 SD 卡引脚

#### 3. 这不会直接破坏我们当前的 WiFi Hosted 项目

当前项目里：

- SD card 走的是 SDMMC Slot0
- ESP32-C6 hosted WiFi 走的是另一套 SDIO 连接（当前日志里是 CLK[18] CMD[19] D0[14] D1[15] D2[16] D3[17]）

也就是说，从当前工程现状看：

- SD 卡 和 C6 Hosted WiFi 不是同一组引脚
- 也不是同一个逻辑通道
- 两者已经在当前项目里同时工作过

这次联调实际上已经证明：

- SD 卡可挂载
- SD 上字体可读
- Hosted WiFi 可扫描 AP
- WiFi 上传服务可起

所以至少在当前板级 BSP 和当前工程配置下：

- 40 MHz SDMMC 不是新增风险项
- 它已经是现状的一部分

#### 4. 真正需要注意的是稳定性验证，而不是“能不能开”

如果后续真要针对 SD 卡做性能优化，应关注的是：

- 长时间读写是否稳定
- 边联网边写 SD 是否有偶发错误
- 上传大文件时速度和错误率如何
- 卡兼容性是否稳定

也就是说：

- 现在不用先去“开高速模式”
- 因为它已经开着
- 更值得做的是：上传大文件、连续读写、并发场景下的稳定性测试

### 11.14 当前阶段最值得继续做的事

基于今天的结果，后续优先级建议如下：

1. 先验证 WiFi 上传服务可把文件稳定写入 /sdcard/BMFONT
2. 定义位图字库包格式（manifest + index + glyph data）
3. 做 PC 端导出工具，把现有字体源文件转换成项目自己的位图字库包
4. 板子端实现“识别已上传字库包 + 显示包信息 + 切换当前字库”
5. 最后再把 LVGL 的文本渲染逐步替换成位图字库加载逻辑

当前不建议：

- 一上来自己设计整套中文字体
- 一上来追求全量中文
- 一上来把上传、解析、切换、热加载、预览全部一次做完

更稳的路线是：

- 先把文件传进去
- 再把文件识别出来
- 再把字形真正显示出来
## 12. 2026-04-21 位图字库联调、滚动性能结论与当前判断

### 12.1 今天实际验证通过的内容

今天围绕 /sdcard/BMFONT 做了单独联调，重点不再是 TTF/TTC 运行时字体，而是位图字库链路本身。

当前已经验证通过的点包括：

- 板子可以从 /sdcard/BMFONT 找到位图字库文件。
- 固件已经可以直接读取位图字库的 index.bin 和 glyph.bin。
- 当前工程里已经补了一个最小位图字库渲染器，不依赖 FreeType 参与这些样例的字形栅格化。
- 位图字库样例可以直接绘制到 LVGL canvas。
- 纯位图字库滚动测试页的拖动流畅度，明显好于之前的“长页面 + SD 上 TTF/TTC + FreeType 动态栅格化”方案。

这次联调最重要的结论是：

- 至少在当前 ESP32-P4 + 10.1 寸屏 + 这套 UI 路线上，位图字库比运行时 TTF/TTC 更适合作为中日文高流畅度显示方案。

### 12.2 当前位图字库路径兼容策略

由于上传链路和 FatFs/8.3 风格命名会带来目录及文件名变化，当前固件没有只写死一种资源布局，而是同时兼容两类路径：

#### 1. 标准目录包

例如：

- /sdcard/BMFONT/gui-ui/zh-CN/20/index.bin
- /sdcard/BMFONT/gui-ui/zh-CN/20/glyph.bin
- /sdcard/BMFONT/gui-ui/ja-JP/20/index.bin
- /sdcard/BMFONT/gui-ui/ja-JP/20/glyph.bin

同时也兼容大写/8.3 风格目录名，例如：

- /sdcard/BMFONT/GUI_UI/ZH_CN/20/INDEX.BIN
- /sdcard/BMFONT/GUI_UI/JA_JP/20/GLYPH.BIN

#### 2. 扁平散文件

例如：

- /sdcard/BMFONT/16cn_index.bin
- /sdcard/BMFONT/16cn_glyph.bin
- /sdcard/BMFONT/20jp_index.bin
- /sdcard/BMFONT/20jp_glyph.bin

也兼容若干 8.3 风格名字，例如：

- 16CN_IND.BIN
- 16CN_GLY.BIN
- JP20_IND.BIN
- JP20_GLY.BIN

这样做的目的很明确：

- 不要求上传链路必须严格保留 PC 端原始目录结构。
- 即使上传后文件被摊平成 /sdcard/BMFONT/*.bin，板子端也仍能尝试识别。

### 12.3 今天的测试页与使用方式

今天已经把当前工程临时切成“纯位图字库滚动基准页”。

这个页面的特点是：

- 启动后不再优先用于 WiFi 联调。
- 不再依赖 /sdcard/FONTS 那套 TTF/TTC 路径来验证滚动性能。
- 页面主体用多组 zh-CN / ja-JP、16 / 20 / 28 px 的位图样例做滚动测试。
- 字形直接从位图包读出后画到 canvas，而不是交给 lv_label + FreeType。

这次的意义在于：

- 它把“UI 长页面卡顿到底来自布局、来自 FreeType、还是来自 SD 位图字库”拆开了。
- 结果显示：纯位图字库页面滚动已经足够顺滑，说明之前的主要压力确实更偏向运行时字体渲染，而不是“ESP32-P4 完全带不动中文界面”。

### 12.4 为什么之前会误以为是整机性能不够

此前长页面里同时叠加了多种开销：

- 页面内容长且持续滚动。
- 中文和日文文本由 TTF/TTC + FreeType 运行时取字形、栅格化、重绘。
- 页面里还有其他说明文字、状态块和周期性刷新逻辑。

在这种情况下，拖动卡顿很容易被误判成：

- ESP32-P4 本身性能不够
- 或者 SD 卡速度不够

但今天把页面改成纯位图字库基准后，实际现象已经说明：

- “滚动很卡”不应简单归因于 P4 性能不够。
- 更准确地说，卡顿主要来自“运行时字体渲染链路 + 长页面重绘”这一组合。

### 12.5 当前关于说明文字、方块字和缺字的判断

今天还联调过一个容易混淆的问题：

- 位图样例本身能正常显示。
- 但页面上方的中文说明文字，最初不是方块，就是缺字，或者干脆不显示。

这里需要分清三种情况：

#### 1. 说明文字仍走普通 lv_label

如果说明文字还在走内置静态字体，那么：

- 字库不全时会出现方块。
- 这不等于位图字库方案失败，只是说明这部分文字根本还没切到位图字库渲染路径。

#### 2. 说明文字已经切到位图字库，但位图包本身缺字

后面把说明文字也切到位图渲染后，方块问题没有了，但又出现了“部分字符直接消失”的现象。

这时根因已经变成：

- 当前位图字库包并不覆盖这些说明文案里的全部汉字。
- 不是 LVGL 方块字问题，而是字库本身缺少对应 glyph。

#### 3. 当前固件对缺字的处理

后续已经把缺字处理从“静默跳过”改成了更容易诊断的形式：

- 若缺少原始目标字形，优先尝试用 ?、*、# 这类 ASCII 备用字形占位。
- 同时在页面上额外给出英文提示，说明有多少个 glyph 被 fallback 替换。

这样后续再看屏幕现象时，更容易判断：

- 是渲染失败
- 还是字库缺字

### 12.6 当前对位图字库路线的更明确判断

截至今天，位图字库方案已经不再只是“一个想法”，而是已经有了比较明确的工程结论：

#### 适合位图字库的场景

- UI 固定字号较少，例如 16 / 20 / 28 px 这类有限规格。
- 中日文文案相对稳定，不要求任意输入法式自由文本。
- 更重视滚动流畅度和首帧确定性，而不是无限灵活的动态排版。

#### 不适合只靠运行时 TTF/TTC 的场景

- 长页面连续滚动
- 大段中日文正文
- 页面里频繁有新字首次出现
- 希望 MCU 端也做到接近手机式连续刷流体验

因此当前更合理的总体方向是：

- 小 fallback 字体继续保留，用于最基础的报错或兜底。
- 常用 UI 中日文文本优先转成位图字库。
- 对必须保留灵活性的场景，再考虑是否继续保留少量运行时 TTF/TTC 字体入口。

### 12.7 SD 卡速率与今天的结论

今天还再次确认了 SD 卡相关配置。

当前工程里，BSP 的 SD 卡已经是：

- SDMMC 4-bit
- High Speed (40 MHz)

因此对于这次位图字库联调来说：

- “要不要把 SD 卡开到 40 MHz”并不是待办项。
- 当前项目本来就已经这么配置。

今天的结果也说明：

- 即使不再继续抠 SDMMC 频率，单就“位图字库替换运行时字体”这一件事，滚动体验也已经有明显改善。

### 12.8 关于今天观察到的 30 FPS 上限

今天联调里还遇到一个现象：

- 页面体感已经更顺了。
- 但 LVGL 的 perf monitor 观察值仍然容易看到“最高约 30 FPS”。

当前不能据此直接下结论说：

- 面板物理刷新率只有 30 Hz

原因有两个：

#### 1. 当前 10.1 寸面板的初始化宏不是 30 Hz 配置

当前使用的 JD9365_800_1280_PANEL_60HZ_DPI_CONFIG(...) 来自 waveshare__esp_lcd_jd9365。其关键时序参数为：

- dpi_clock_freq_mhz = 80
- h_size = 800
- v_size = 1280
- hsync_back_porch = 20
- hsync_pulse_width = 20
- hsync_front_porch = 40
- vsync_back_porch = 10
- vsync_pulse_width = 4
- vsync_front_porch = 30

按这组参数估算，刷新率并不是 30 Hz 量级。

#### 2. LVGL 的 perf monitor 本身会受显示刷新定时器周期钳制

LVGL 的 lv_sysmon 在计算显示出来的 FPS 时，会把值限制到：

- 1000 / disp_refr_period

也就是说：

- 如果当前 monitor 看到的 display refresh timer 仍是 33 ms
- 那么屏幕上显示的 FPS 本来就不可能超过 30

因此截至今天，对这个问题的最稳妥判断是：

- “最高只看到 30 FPS” 目前更像统计口径或定时器口径问题。
- 它还不能单独证明“LCD 物理只跑 30”。

这个问题后面若要彻底坐实，应单独做“真实刷新计数”诊断，而不是只盯着 perf monitor 数字。

### 12.9 当前遗留问题

虽然位图字库路线已经明显更靠谱，但今天仍有几项遗留问题：

1. 位图字库覆盖范围还不完整，说明文案里仍可能出现缺字。
2. 当前只验证了样例与基准页，尚未把整套正式 UI 文本全部切到位图字库。
3. 还没有做 glyph bitmap RAM cache，当前仍属于“能工作、能看出优势，但还能继续优化”的阶段。
4. 30 FPS monitor 上限的根因还没有做最终闭环验证。

### 12.10 当前最值得继续做的事

结合今天的结果，后续优先级建议更新为：

1. 继续完善 /sdcard/BMFONT 路径下的位图字库包生成与上传链路。
2. 优先让正式 UI 里的常用中日文标题、菜单、状态标签逐步切到位图字库。
3. 对位图字库包缺字问题，补一个更明确的导出字符集清单，不要完全靠实机碰运气。
4. 若还要继续优化交互流畅度，可在板子端加入常用 glyph bitmap 的 RAM cache。
5. 把“真实刷新率诊断”和“LVGL perf monitor 口径问题”单独拆成一项显示链路任务，不要和字体问题继续混在一起判断。
