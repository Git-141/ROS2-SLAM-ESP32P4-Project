## 10. 2026-04-08 SD 字体与竖屏长页面联调补充

### 10.1 今天已验证通过的内容

- SD 卡方案已经跑通，当前更稳的选择是 `32GB + MBR + 单分区 + FAT32`
- ESP32-P4 可以成功挂载 `/sdcard`
- 固件可以在启动时把字体写入 SD 卡
- 运行时 FreeType 可以从 `/sdcard/FONTS` 加载字体
- UI 已切回竖屏默认方向
- 页面已经改成长页面，整体可以上下滑动

今天实机已经确认过的关键日志特征：

- `sdcard mounted at /sdcard`
- `sd asset already present: /sdcard/FONTS/CN_UI.TTF`
- `ui_theme: loaded runtime font: /sdcard/FONTS/CN_UI.TTF`
- `ui_theme: loaded runtime font: /sdcard/FONTS/JP_UI_R.TTC`
- `ui_theme: loaded runtime font: /sdcard/FONTS/SYMBOLS.TTF`

### 10.2 SD 卡与文件系统方面的结论

最初 64GB 卡的主要问题不是容量，而是常见格式为 `exFAT`。
当前工程挂载 SD 使用的是 FAT 路线，因此实操上 `32GB FAT32` 更省事、更稳。

当前建议继续保持：

- 分区表：`MBR`
- 文件系统：`FAT32`
- 字体目录：`/sdcard/FONTS`

### 10.3 中文字体写入失败的根因

一开始中文字体写入失败，不是因为 SD 卡没挂载，而是因为文件名超出了当前 FatFs 允许范围。

实际问题是：

- 当前工程没有开启长文件名支持
- 使用了 `UI_FALLBACK.TTF` 这种超出 8.3 风格的文件名
- 结果 `fopen()` 直接失败

后续已经改成短文件名：

- 中文：`CN_UI.TTF`
- 符号：`SYMBOLS.TTF`

这里得到的经验很明确：

- 只要当前 FatFs 配置不变，文件名就应尽量保持短 ASCII 名
- 不要直接用中文名、日文名、超长名

### 10.4 为什么日语能显示但中文最初不行

这个阶段很容易误判。

当时日语能出来，不代表 SD 字体已经成功生效。更可能是：

- 内置字体本身覆盖了一部分 CJK / 日文字符
- 但中文主字体实际上还没真正写进去或加载成功

所以后面判断字体链路是否生效时，必须同时看：

- 屏幕现象
- 顶部状态提示
- 串口日志

不能只看“某些字出来了”。

### 10.5 中文字体触发 FreeType 栈保护错误的根因

中文字体真正开始加载之后，系统出现过重启循环。

关键 panic 特征：

- `Stack protection fault`
- 落点在 FreeType 的 `cf2_interpT2CharString`

说明问题不在 SD 挂载，不在字体路径本身，而在字体格式和任务栈。

当时的根因组合是：

- 中文字体先用了 `Noto Sans SC (TrueType).otf`
- 这条 OTF/CFF 路线在当前 FreeType 场景下更吃栈
- `lvgl` 任务栈太小

后续处理改成：

- 放弃当前那份 OTF 中文字体
- 改用 `simhei.ttf` 作为中文主字体并内嵌到固件
- 运行时落盘为 `/sdcard/FONTS/CN_UI.TTF`
- 同时把 `lvgl` 任务栈增大，并尝试放入 `PSRAM`

当前经验是：

- 这套平台上，中文字体优先尝试 `TTF/TTC`
- 不要默认优先上 `OTF/CFF`

### 10.6 当前实际采用的字体策略

目前这条链路更接近验证版，但已经是可工作的方向：

- 中文主字体：`CN_UI.TTF`
- 日文字体：`JP_UI_R.TTC`
- 符号字体：`SYMBOLS.TTF`
- Emoji：`EMOJI.TTF`

当前逻辑是：

- 固件内嵌中文和符号字体
- 启动后若 SD 挂载成功，则把它们写到 `/sdcard/FONTS`
- 若文件已存在且大小一致，则跳过重复写入
- 运行时再从 SD 路径加载

这种做法的优点是：

- 开发阶段不需要每次手动拔卡拷文件
- 可以先验证“板子自己写卡 + 自己读卡”的闭环

## 11. 2026-04-09 字体链路瘦身、分页阅读页与性能判断

> 本节覆盖第 10 节里与“固件内嵌字体、长页面滚动测试”有关的后续变化。涉及当前实现时，以本节为准。

### 11.1 今天实际完成的修改

1. 去掉了“把字体编进固件再写入 SD 卡”的中间链路。
2. 当前固件只负责挂载 `/sdcard` 并检查字体文件是否存在，不再负责把字体写回 SD 卡。
3. UI 运行时继续直接从 `/sdcard/FONTS` 读取字体。
4. 阅读页不再做整页连续滚动测试，已经改成分页阅读模式。
5. 页面正文里所有“测试说明、分辨率提示、SD 说明、滚动说明”都已经删除，只保留日文正文和中文翻译。
6. FreeType 字形缓存从 `256` 提高到 `1024`，用于降低重复取字形时的卡顿。

当前字体路径仍是：

- `/sdcard/FONTS/CN_UI.TTF`
- `/sdcard/FONTS/JP_UI_R.TTC`
- `/sdcard/FONTS/SYMBOLS.TTF`
- `/sdcard/FONTS/EMOJI.TTF`

### 11.2 今天遇到的核心问题

#### 11.2.1 为什么固件曾经非常大

根因不是正文文本，而是字体文件被 `EMBED_FILES` 编进了 app 镜像。

当时的链路实际上是：

1. 字体先内嵌到固件。
2. 开机后再把内嵌字体写入 SD。
3. UI 再从 SD 读取这些字体。

这会导致：

- 固件体积非常大。
- SD 已经有字体时，这条链路仍然显得重复。
- 用户容易误以为“字体既然已经在 SD 里，为什么 bin 还是这么大”。

后续已经处理为：

- 删除 `EMBED_FILES`
- 删除启动写卡逻辑
- 仅保留“挂载 SD + 检查文件 + 运行时加载字体”

处理后，固件体积从接近 `14 MB` 量级降到了约 `1.6 MB` 量级，app 分区重新恢复为大量空余。

#### 11.2.2 为什么长文滚动会明显卡顿

滚动卡顿的主因不是“SD 卡慢”这一个因素，而是下面几项叠加：

- 使用了 `SD + FreeType` 运行时字体
- 页面里有大段中文和日文正文
- 触摸拖动时需要持续重排文本、查字形、栅格化、重绘
- 若字形缓存命中率不高，就会不断回到字体文件取新字形

实机现象是：

- 早期 perf 可能从几十毫秒升到上千毫秒
- 调整后已降到大约 `300 ms` 左右
- 体感上仍有明显卡顿，不适合作为“像手机一样刷流”的方案

#### 11.2.3 为什么“下一页”慢、“上一页”快

当前分页版里，这很大概率就是缓存命中差异造成的，属于符合预期的现象。

更具体地说：

- 点击“下一页”时，往往要首次显示一批新的汉字/日文字形。
- 这些字形如果还没被 FreeType/LVGL 缓存过，就要现场从字体文件读取并栅格化。
- 点击“上一页”时，前一页刚刚显示过，相关字形仍很可能还留在 RAM 缓存里，所以会更快。

结论：

- `上一页` 快，通常是因为前一页字形和布局结果刚刚用过。
- `下一页` 慢，通常是因为遇到了新的字形集合。

这个缓存是运行时 RAM 缓存，不是写回 SD 的持久缓存。

因此下一次重新上电时：

- SD 卡里的字体文件当然还在。
- 但 FreeType / LVGL 的字形缓存会重新开始建立。
- 也就是说，首次翻到新页时仍然可能再次出现“第一次慢、回看快”的现象。

### 11.3 今天采用的解决方式

1. 删掉固件内嵌字体，只保留 SD 运行时字体。
2. 删掉启动写 SD 卡逻辑，避免继续增大镜像体积。
3. 清理阅读页里的调试说明文本，减少干扰。
4. 把整页滚动阅读改成分页阅读，避免在拖动过程中持续重排和重绘整页长文。
5. 提高 FreeType 字形缓存数量，降低重复页面切换时的栅格化压力。

### 11.4 当前更合理的判断

如果目标是：

- 在 MCU 上显示中日文长文
- 页面内容主要用于查看说明、记录、手册、日志摘要
- 不追求手机社交软件式连续滚动

那么“分页阅读”比“长页面连续滚动”更现实，也更稳定。

如果目标是：

- 真正做到像手机那样顺滑地刷长文流
- 同时又使用大段中日文和运行时 TTF/TTC 字体

那对当前这块板子和这套 GUI 路线来说，成本会明显偏高，不应作为近期主目标。

### 11.5 下一步建议

建议按优先级这样推进：

1. 先稳定分页阅读页，确认分页切换、字体显示、触摸命中都可接受。
2. 观察“首次下一页较慢”是否只发生在首次访问新页时。
3. 如果要进一步优化翻页首帧，可考虑做“字形预热”或“预翻页预渲染”。
4. 如果后面要做真正的阅读器，不建议继续押注“长文连续滚动 + SD 上动态字体”。
5. 若目标是实用型阅读器，应优先考虑位图字库或离线预渲染方案。

### 11.7 本机如何用 PowerShell 稳定写文件

如果需要用 PowerShell 直接改文件，当前更稳的方式不是 `Set-Content`，而是：

1. 先 `Get-Content -Raw` 读全文。
2. 用 `.Replace(...)` 做精确替换，或用 here-string 准备整段内容。
3. 用 `[System.IO.File]::WriteAllText(...)` 写回。
4. 显式指定 `UTF8Encoding($false)`，避免 BOM 带来的额外问题。

已验证可用的模板：

```powershell
$path = 'components\terminal_gui_app\src\ui\ui_shell.c'
$content = Get-Content -Path $path -Raw
$content = $content.Replace('旧内容', '新内容')
$utf8NoBom = New-Object System.Text.UTF8Encoding($false)
[System.IO.File]::WriteAllText((Resolve-Path $path), $content, $utf8NoBom)
```

如果是追加整段文档，可直接这样写：

```powershell
$path = 'D:\Vscode\ROS2\ROS2-SLAM-ESP32P4-Project\docs\esp32p4.md'
$append = @"
这里放要追加的 Markdown 内容。
"@
$old = Get-Content -Path $path -Raw
$utf8NoBom = New-Object System.Text.UTF8Encoding($false)
[System.IO.File]::WriteAllText($path, $old + $append, $utf8NoBom)
```

这么做的原因是：

- 可控性更高，适合大段文本。
- 不容易因为换行或编码差异把文件写乱。
- 这次会话里已经多次用这种方式稳定修改源码和文档。

### 11.8 位图字库方案的想法

当前已经可以明确区分两条路线：

#### A. 动态栅格化字体

即当前方案：

- SD 卡里放 `TTF/TTC`
- 运行时由 FreeType 按需查找并栅格化字形

优点：

- 灵活
- 一个字体文件可覆盖多个字号
- 文本仍然是文本，适合动态内容

缺点：

- MCU 上 CPU 压力大
- 长文 + CJK + 高频翻页/滚动时性能代价明显
- 首次遇到新字时很容易卡顿

#### B. 位图字库

思路是：

- 提前把固定字号、固定字重的常用字生成位图
- 运行时直接读取位图并显示
- 可以把位图字库放在 SD 卡里，不一定要编进固件

优点：

- 比动态栅格化轻得多
- 更适合 MCU
- 翻页或滚动时性能更可控

缺点：

- 字号不灵活
- 字集一大时，字库体积会明显增长
- 需要自己设计索引、缓存和加载方式

当前更值得考虑的落地方向是：

1. 固定字号位图字库 + SD + 分页阅读
2. 整页预渲染图片 + SD + 分页阅读

如果目标是“实用型 MCU 阅读器”，这两条路线都比“SD 上 TTF/TTC 动态栅格化长文流”更靠谱。

### 11.9 当前结论

截至 2026-04-09，当前项目在阅读场景下更适合的方向已经比较明确：

- 先以分页阅读为主，不再强求手机式长文滚动。
- 字体继续直接从 SD 读取，不再重复写卡，也不再把大字体编进固件。
- 如果后续仍觉得翻页首帧偏慢，应优先考虑预热缓存、位图字库或预渲染，而不是继续死磕连续滚动。
